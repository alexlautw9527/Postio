# Postio MVP 設計規格

**日期**：2026-07-08
**狀態**：待審查
**Tagline**：Write once. Publish anywhere.

## 1. 產品定位

Postio 是一個純靜態單頁網站，把 Markdown 轉換成適合貼上 Threads 的 Unicode 純文字。MVP 只針對 Threads，目標受眾是台灣 Threads 社群，介面為繁體中文。

核心體驗（30 秒內完成）：

1. 貼上 Markdown
2. 即時預覽轉換結果（超過 500 字自動分段成 thread）
3. 逐篇複製
4. 貼到 Threads

## 2. 功能範圍

### Must Have

- Markdown 輸入區（textarea），輸入即轉換，無需按鈕觸發
- 核心轉換規則（見第 3 節），含巢狀清單與 checkbox
- 超過 500 字自動分段成多篇，篇尾 `(n/m)` 編號
- 每篇一張卡片：內文預覽、字數、獨立複製按鈕與「已複製」回饋
- 主題切換：至少兩種預設主題（▍直線、◆菱形）
- 繁體中文介面

### Nice to Have（本次一併實作）

- `localStorage` 保存草稿與主題選擇，重新整理不遺失
- 預設模板：首次進入（無既有草稿）時，編輯器預先載入一份範例 Markdown，內容同時是簡單教學說明（使用步驟與支援格式一覽），一進頁面就能看到轉換結果；清空編輯器後顯示「載入範例」按鈕可隨時載回

### Future（明確不做）

- 自訂符號主題（MVP 只有預設主題切換）
- 其他平台（X、LinkedIn 等）
- Threads API 直接發文
- 表格、程式碼區塊的完整格式支援
- 手動調整分段點

### 明確刪除（連 Future 都不是）

- 帳號、雲端同步、AI、模板（產品原則）
- 一鍵複製全部：Threads 需逐篇貼上，此功能誤導使用者
- 編輯器功能（工具列、語法高亮）：使用者在他處寫作完成後才過來

## 3. 轉換規則

### 3.1 行內樣式

| Markdown | 輸出 | 說明 |
|---|---|---|
| `**粗體**` | 𝗯𝗼𝗹𝗱 | Mathematical Sans-Serif Bold |
| `*斜體*` | 𝘪𝘵𝘢𝘭𝘪𝘤 | Mathematical Sans-Serif Italic |
| `***粗斜體***` | 𝙗𝙤𝙡𝙙 𝙞𝙩𝙖𝙡𝙞𝙘 | Sans-Serif Bold Italic |
| `[文字](url)` | 文字 (url) | 純 URL 直接保留 |
| `` `inline code` `` | 去掉反引號保留文字 | 優雅降級 |

**已知限制**：Unicode 假字型只涵蓋拉丁字母與數字（A–Z、a–z、0–9），中文字無對應碼位、無法變粗。中文內容的視覺層級完全依靠符號前綴與空行——此為標題符號主題存在的核心理由。假字型對螢幕閱讀器不友善，MVP 接受此取捨。

### 3.2 區塊元素

| Markdown | 輸出 |
|---|---|
| `# H1` | 主題 H1 前綴 + Unicode 粗體 + 前後空行 |
| `## H2` | 主題 H2 前綴 + Unicode 粗體 + 前後空行 |
| `### H3`（H4–H6 視同 H3） | 主題 H3 前綴 + Unicode 粗體 + 前後空行 |
| `---` | 主題定義的分隔線（如 ───────） |
| `> 引用` | 以 ❝ ❞ 包裹：`❝ 引用內容 ❞`（多行引用整段包裹，首尾各一個引號，內部換行保留） |
| `![alt](src)` | 輸出 alt 文字；alt 為空則整個略過 |

### 3.3 清單（含巢狀與 checkbox）

解析需啟用 `remark-gfm`（task list 為 GFM 語法）。

**無序清單**：依巢狀層級使用不同項目符號，每層縮排 3 個半形空格：

```
• 第一層
   ◦ 第二層
      ▪ 第三層（更深層級沿用 ▪）
```

**有序清單**：保留數字編號，巢狀層縮排並沿用該層自己的編號：

```
1. 第一層
   1. 第二層
```

**Checkbox（task list）**：

| Markdown | 輸出 |
|---|---|
| `- [ ] 待辦` | ⬜ 待辦 |
| `- [x] 完成` | ✅ 完成 |

採用 emoji 對（⬜／✅）而非 ☐／☑，因為兩者在行動裝置上寬度一致、辨識度高。checkbox 可與巢狀清單組合（縮排規則同無序清單，符號替換為 ⬜／✅）。

**待驗證假設**：Threads 是否保留行首空格（巢狀縮排依賴此行為）。實作初期先在 Threads 實測；若被剝除，改用不會被剝除的空白字元（如 U+2002 EN SPACE）作為縮排單位。

### 3.4 主題

主題定義 = 標題前綴（H1/H2/H3）+ 分隔線樣式。MVP 內建兩種：

| | H1 | H2 | H3 | 分隔線 |
|---|---|---|---|---|
| 直線 | ▍ | ▎ | ▏ | ─────── |
| 菱形 | ◆ | ◇ | ▸ | ─────── |

主題結構設計為資料驅動（`themes.ts` 中的物件），未來加主題或開放自訂只需擴充資料。

## 4. 分段演算法

1. 先將整篇 Markdown 轉換為帶樣式的純文字「區塊」序列（標題、段落、清單項、分隔線各為一個區塊）
2. 貪婪裝箱：依序將區塊塞入當前貼文，放不下即開新篇
3. 每篇上限 = 500 −「篇尾編號保留字數」（12 字，含前置空行，足以容納三位數篇數如 `(999/999)`）；先以 500 全額嘗試裝箱，結果為單篇即直接採用（單篇不編號、不需保留）
4. 單一區塊本身超過上限：在句子邊界（。！？．!?. 之後）切分；單句仍超長才硬切並在該卡片顯示警告
5. **標題不落單**：若標題（含連續多個標題）會成為某篇的最後區塊，整組推到下一篇
6. **清單不切開**：連續的清單項視為同一單位；當前篇塞不下且整組放得進新的一篇時，整組移到下一篇，只有清單本身超過單篇容量才允許在項目邊界切開
7. 字數以 `Intl.Segmenter` 的 grapheme 計算（emoji、組合字元不會爆字數）
8. 總篇數為 1 時不加編號；多篇時每篇篇尾加 `(n/m)`
9. 編號字數計入 500 字上限

## 5. 架構

### 5.1 技術選型

- Vite + React + TypeScript
- Markdown 解析：`remark-parse` + `remark-gfm`（標準 mdast AST，不自寫 parser）
- 測試：Vitest
- 部署：靜態 hosting（Vercel 或 GitHub Pages）
- 無後端、無帳號、無任何網路請求（除載入頁面本身）

### 5.2 目錄結構

```
src/
  core/            ← 純 TypeScript，零 React 依賴
    unicode.ts     字元映射表（bold / italic / boldItalic）
    themes.ts      主題定義（標題前綴、分隔線）
    render.ts      mdast AST → 帶樣式的純文字區塊序列
    split.ts       區塊序列 → Post[]（分段演算法）
    index.ts       convert(markdown, themeId): Post[]
  components/
    Editor.tsx       Markdown 輸入區
    PostCard.tsx     單篇卡片（內文、字數、複製按鈕）
    ThemePicker.tsx  主題切換
  App.tsx
```

### 5.3 資料流

單向：textarea `onChange` → `convert()` → `Post[]` → 卡片列表。狀態只用 `useState`（輸入內容、主題選擇），不引入狀態管理庫。文字量小，每次輸入直接重算，不做 debounce。

`Post` 型別：

```ts
interface Post {
  text: string        // 最終可複製的文字（含編號）
  charCount: number   // grapheme 數
  overflow: boolean   // 硬切警告用
}
```

## 6. 錯誤處理與邊緣案例

- 空輸入：預覽區顯示引導文案與「載入範例」
- 剪貼簿 API 失敗（非 HTTPS、舊瀏覽器）：fallback 為自動全選該篇文字，提示使用者手動複製
- 巢狀行內格式（粗體中包斜體）：以粗斜體字元輸出
- 純 emoji、零寬字元輸入：grapheme 計數保證不誤判字數
- 無效 Markdown：remark 本身容錯，永不報錯，最差情況原樣輸出

## 7. 測試策略

`core/` 全部單元測試（Vitest）：

- Unicode 映射：大小寫字母、數字、非拉丁字元穿透（中文原樣輸出）
- 每種元素的轉換：六項核心 + 巢狀清單各層級 + checkbox 勾選/未勾選 + 引用（單行/多行）+ 降級元素
- 分段邊界：499/500/501 字、標題不落單、單區塊超長的句子切分、單句超長硬切、編號字數計入上限、單篇不編號
- 主題切換影響前綴與分隔線輸出

UI 元件不寫測試（MVP 階段），以手動驗證為主。

## 8. 驗收標準

1. 貼上含標題、粗體、巢狀清單、checkbox 的 Markdown，1 秒內看到分段後的預覽
2. 每篇卡片字數 ≤ 500，複製後貼到 Threads 格式正確
3. 超過 500 字的文章正確分段，編號正確，標題不落單
4. 重新整理頁面，草稿與主題選擇仍在
5. 首次開啟頁面（無草稿）即看到範例教學與其轉換結果，30 秒內能完成一次完整流程
